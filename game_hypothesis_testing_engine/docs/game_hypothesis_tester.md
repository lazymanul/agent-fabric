# Игровой движок валидации гипотез (Game Hypothesis Testing Engine)

## Описание

Игровой движок валидации гипотез — это система автоматизированной проверки технологических гипотез с использованием геймифицированного подхода. Движок декомпозирует гипотезы на атомарные утверждения, проверяет их на основе базы знаний, генерирует контраргументы ("Адвокат дьявола") и оценивает общую обоснованность гипотезы.

Основная цель — объективизация процесса принятия решений при оценке технических гипотез для обогащения руды.

## Модели данных

### Hypothesis (Гипотеза)

Основная сущность — гипотеза на валидацию.

```python
@dataclass
class Hypothesis:
    id: str                           # Уникальный идентификатор (HYP-TOF-01)
    title: str                        # Краткое название
    description: str                  # Полное описание
    claims: List[Claim]               # Атомарные утверждения
    challenges: List[Challenge]       # Контраргументы
    total_score: int                  # Набранные очки
    max_possible_score: int           # Максимально возможные очки
    status: HypothesisStatus          # Итоговый статус
```

### Claim (Утверждение)

Атомарное утверждение, на которое декомпозируется гипотеза.

```python
@dataclass
class Claim:
    id: str                           # Идентификатор (HYP-TOF-01-C01)
    text: str                         # Текст утверждения
    status: ClaimStatus               # Статус проверки
    evidence: List[Evidence]          # Найденные доказательства
    points_earned: int                # Заработанные очки
    max_points: int                   # Максимальные очки (обычно 15)
    counterarguments: List[str]       # Атаки контраргументов
    rebuttals: List[str]              # Возражения на атаки
```

### Evidence (Доказательство)

Фрагмент данных из базы знаний, подтверждающий или опровергающий утверждение.

```python
@dataclass
class Evidence:
    source: str                       # Источник (файл, статья)
    snippet: str                      # Цитата/фрагмент
    data_point: Optional[Dict]        # Конкретная цифра/факт
    confidence: float                 # Уверенность (0.0 - 1.0)
```

### Challenge (Контраргумент)

Атака от "Адвоката дьявола" и защита от неё.

```python
@dataclass
class Challenge:
    round_number: int                 # Номер раунда
    attack_text: str                  # Текст контраргумента
    defense_text: Optional[str]       # Текст возражения
    resolved: bool                    # Разрешен ли спор
    points_delta: int                 # Изменение очков
```

### Статусы

**ClaimStatus** — статус проверки утверждения:
- `VERIFIED` — подтверждено (уверенность ≥ 90%)
- `PARTIALLY_VERIFIED` — частично подтверждено (уверенность ≥ 70%)
- `UNVERIFIED` — не подтверждено (уверенность < 70%)
- `CONTRADICTED` — опровергнуто

**HypothesisStatus** — итоговый статус гипотезы:
- `APPROVED_FOR_PILOT` — готова к пилотным испытаниям (≥ 80%)
- `APPROVED` — одобрена (≥ 60%)
- `NEEDS_REVISION` — требует доработки (≥ 40%)
- `REJECTED` — отклонена (< 40%)

## Принцип работы

### Шаг 1: Загрузка данных

```python
parser = DataParser(data_dir="data")
context = parser.parse_tailings_excel("Хвосты ТОФ_2.xlsx")
hypotheses_data = parser.parse_hypotheses_docx("Гипотезы ТОФ.docx")
```

**DataParser** выполняет:
- Парсинг Excel-файлов с данными о хвостах обогащения (гранулометрический состав, минералогия, массы)
- Парсинг DOCX-файлов с гипотезами (поддержка таблиц Word, параграфов и regex-fallback)
- Формирование контекста — структурированного словаря с данными для поиска доказательств

### Шаг 2: Инициализация компонентов

```python
llm_adapter = LLMAdapter()
game_engine = ValidationGameEngine(context, llm_adapter)
reporter = ReportGenerator(output_dir="output")
```

### Шаг 3: Валидация гипотезы

Основной метод `game_engine.validate(hypothesis_data)` запускает игровой процесс валидации.

#### РАУНД 1: Декомпозиция и фактологическая проверка

```
┌─────────────────────────────────────────────────────────────────────┐
│                    РАУНД 1: ПРОВЕРКА ФАКТОВ                         │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  1. LLM декомпозирует гипотезу на атомарные утверждения (claims)   │
│                                                                     │
│  2. Для каждого claim:                                              │
│     ┌─────────────────────────────────────────────────────────┐    │
│     │  a. EvidenceFinder ищет доказательства в контексте      │    │
│     │     - Паттерн-матчинг по ключевым словам                │    │
│     │     - Извлечение данных из контекста                    │    │
│     │                                                         │    │
│     │  b. Рассчитывается средняя уверенность (confidence)     │    │
│     │                                                         │    │
│     │  c. Назначается статус и начисляются очки:              │    │
│     │     ┌──────────────────────────────────────────────┐    │    │
│     │     │ confidence ≥ 90% → VERIFIED, +max_points     │    │    │
│     │     │ confidence ≥ 70% → PARTIALLY_VERIFIED, +60%  │    │    │
│     │     │ confidence < 70% → UNVERIFIED, +0            │    │    │
│     │     │ нет доказательств → UNVERIFIED, +0           │    │    │
│     │     └──────────────────────────────────────────────┘    │    │
│     └─────────────────────────────────────────────────────────┘    │
│                                                                     │
│  3. Обновляется total_score и max_possible_score гипотезы          │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**Пример декомпозиции:**

Гипотеза: *"Донастройка скорости вращения классификаторов"*

Декомпозиция на claims:
1. "В хвостах присутствуют значительные потери ценных компонентов в классе -10 мкм из-за перешлиховки"
2. "Введение дополнительной классификации позволит отсечь тонкие классы из цикла измельчения"
3. "Оптимизация скорости вращения классификаторов повысит эффективность разделения"

#### РАУНД 2: Контраргументы (Адвокат дьявола)

```
┌─────────────────────────────────────────────────────────────────────┐
│              РАУНД 2: КОНТРАРГУМЕНТЫ И ЗАЩИТА                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  Для каждого VERIFIED или PARTIALLY_VERIFIED claim:                 │
│                                                                     │
│  1. LLM генерирует контраргумент (атака)                            │
│     ┌─────────────────────────────────────────────────────────┐     │
│     │  Пример: "Грохочение на 10 мкм в мокрых процессах       │     │
│     │  неэффективно — грохоты забьются из-за высокой          │     │
│     │  влажности. Требуется дорогостоящее обезвоживание."     │     │
│     └─────────────────────────────────────────────────────────┘     │
│                                                                     │
│  2. LLM генерирует возражение (защита)                              │
│     ┌─────────────────────────────────────────────────────────┐     │
│     │  Пример: "Принимаю замечание. Вместо грохотов           │     │
│     │  предлагается использовать батарею гидроциклонов        │     │
│     │  малых диаметров (ГЦМД), которые эффективно работают    │     │
│     │  с мокрыми шламами."                                    │     │
│     └─────────────────────────────────────────────────────────┘     │
│                                                                     │
│  3. Начисляются бонусные очки за успешную защиту (+2)              s │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### Шаг 4: Подсчет итоговой оценки

```python
score_percentage = (total_score / max_possible_score) × 100%
```

**Определение статуса:**

| Процент | Статус | Интерпретация |
|---------|--------|---------------|
| ≥ 80% | `APPROVED_FOR_PILOT` | Гипотеза готова к пилотным испытаниям |
| ≥ 60% | `APPROVED` | Гипотеза одобрена, нужны доп. испытания |
| ≥ 40% | `NEEDS_REVISION` | Гипотеза требует доработки |
| < 40% | `REJECTED` | Гипотеза отклонена |

### Шаг 5: Генерация рекомендаций

На основе статуса и контраргументов генерируются рекомендации:

- Для `APPROVED_FOR_PILOT`: "Гипотеза готова к пилотным испытаниям на одном из потоков"
- Для `APPROVED`: "Гипотеза одобрена, но требует дополнительных лабораторных испытаний"
- Для `NEEDS_REVISION`: "Гипотеза требует доработки. Слабые места выявлены в Раунде 2"
- Для `REJECTED`: "Гипотеза отклонена. Требуется фундаментальный пересмотр подхода"

Дополнительные рекомендации добавляются на основе контраргументов:
- Упоминание "капитальных затрат" → "Подготовить детальный расчет ROI"
- Упоминание "нагрузки" → "Провести аудит текущей загрузки оборудования"
- Упоминание "грохотов" → "Рассмотреть альтернативу: батарею ГЦМД"

### Шаг 6: Генерация дорожной карты

Формируется поэтапный план проверки:

1. **Лабораторный** (2 недели)
   - Грохо-фракционный анализ проб хвостов
   - Минералогический анализ класса -10 мкм

2. **Пилотный** (1 месяц)
   - Установка пилотной батареи ГЦМД
   - Сбор технологических проб
   - Оценка эффекта на извлечение

3. **Промышленный** (3 месяца)
   - Масштабирование на всю ТОФ
   - Мониторинг KPI
   - Корректировка параметров

### Шаг 7: Оценка рисков

Генерируется таблица рисков:

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| Забивание мокрых грохотов | Высокая | Крит. | Замена на батарею ГЦМД |
| Увеличение нагрузки на мельницы | Средняя | Среднее | Оптимизация футеровки |
| Капитальные затраты | Высокая | Высокое | Расчет ROI |

### Шаг 8: Генерация отчетов

```python
reporter.generate_json(result)      # JSON для программной обработки
reporter.generate_markdown(result)  # Markdown паспорт для человека
reporter.generate_jira_json(result) # Задача для Jira/YouTrack
```

## Система оценки

### Начисление очков

| Действие | Очки |
|----------|------|
| Claim подтвержден (VERIFIED) | +15 (max_points) |
| Claim частично подтвержден (PARTIALLY_VERIFIED) | +9 (60% от max) |
| Claim не подтвержден (UNVERIFIED) | +0 |
| Успешная защита от контраргумента | +2 |

### Формула итоговой оценки

```
score_percentage = Σ(points_earned) / Σ(max_points) × 100%
```

## Форматы отчетов

### JSON (`{id}_result.json`)

Полный результат для программной обработки:
- Данные гипотезы (id, title, status, score)
- Список claims с доказательствами и контраргументами
- Рекомендации, дорожная карта, риски
- Полный game_log

### Markdown (`{id}_passport.md`)

Паспорт гипотезы для человека:
- Описание гипотезы
- Проверенные утверждения с эмодзи-статусами
- Отработанные контраргументы
- Рекомендации
- Дорожная карта
- Таблица рисков

### Jira JSON (`{id}_jira.json`)

Задача для трекера:
- Project: RND
- IssueType: Research Hypothesis
- Priority на основе score_percentage
- Labels: hypothesis-validation, game-engine
- Custom fields для score и status

## Пример использования

```python
from core.parsers import DataParser
from core.game_engine import ValidationGameEngine
from core.llm_adapter import LLMAdapter
from core.reporter import ReportGenerator

# 1. Загрузка данных
parser = DataParser(data_dir="data")
context = parser.parse_tailings_excel("Хвосты ТОФ_2.xlsx")
hypotheses_data = parser.parse_hypotheses_docx("Гипотезы ТОФ.docx")

# 2. Инициализация
llm_adapter = LLMAdapter()
game_engine = ValidationGameEngine(context, llm_adapter)
reporter = ReportGenerator(output_dir="output")

# 3. Валидация каждой гипотезы
for hyp_data in hypotheses_data:
    result = game_engine.validate(hyp_data)
    
    # Генерация отчетов
    reporter.generate_json(result)
    reporter.generate_markdown(result)
    reporter.generate_jira_json(result)
```
